## 三、核心结论

### 结论 1：问题不只是缺少 9.3 的 wasm 或 `empty_bin.ts`

`x2t.wasm`、`empty_bin.ts`、`sdkjs/web-apps` 是三类不同资产，不能混为一谈。

| 层级 | 作用 | 当前风险 |
|------|------|----------|
| `sdkjs/web-apps` | 编辑器 UI、sdkjs 渲染和打开文档入口 | 9.3 默认进入 server/desktop 架构，不等价于 7.x standalone |
| `x2t.wasm` | 格式转换器，例如 OOXML 与 OnlyOffice 内部格式之间转换 | 需要确认是否与 9.3 内部格式同代 |
| `empty_bin.ts` | 新建空文档模板 | 当前是旧的 `DOCY;v5` / `XLSY;v2` / `PPTY;v1`，不能默认被 9.3 兼容 |

即使补齐 9.3 的 `x2t.wasm` 和空模板，也只能解决输入格式的一部分；9.3 编辑器初始化路径仍然需要被正确激活。

### 结论 2：`asc_openDocumentFromBytes` 不能继续当作万能打开入口

新文档对 `a9c(data, "DOCY")` 的分析很关键。9.3 的 `asc_openDocumentFromBytes` 会把输入包装成 `WYc`，并用 `a9c` 判断是否是 DOCY byte signature。

已记录的判断：

| 输入 | `a9c(data, "DOCY")` | 风险 |
|------|---------------------|------|
| JS string：`DOCY;v5;7372;...` | `false` | string 索引返回字符，不是 char code number |
| 解码后的 7372 bytes | `false` | 丢掉了 `DOCY;v5;7372;` envelope，前 4 bytes 不是 `DOCY` |
| `TextEncoder().encode("DOCY;v5;...")` | `true` | 只能通过 signature，不代表后续 9.3 能解析旧内部格式 |
| 原始 `.docx` ZIP bytes | `false` | 不是 DOCY 分支输入 |

因此，当前看到的 `Asc.c_oAscError.ID.ConvertationOpenFormat` / generic `errorInconsistentExt` 与源码逻辑一致，不应继续盲目尝试 string 和 decoded bytes。

### 结论 3：当前代码与调试文档存在实现偏差

`2026-06-14-desktop-mock-9.3.0-debug.md` 里写到：

- `api.js` 已 patch：`ver=''`
- `api.js` 已 patch：`parentOrigin="file://"`
- runtime 能看到 `[DE] MOa patched -> BRj path active`

但当前文件状态显示：

| 文件 | 当前状态 | 影响 |
|------|----------|------|
| `public/web-apps/apps/api/documents/api.js` | `_config.parentOrigin = window.location.origin` | 与文档记录的 `file://` 不一致 |
| `public/web-apps/apps/api/documents/api.js` | `const ver = '/9.3.0-{{HASH_POSTFIX}}'` | 仍会生成带版本前缀的 editor URL |
| `vite.config.ts` | `onlyofficeVersionRewrite()` 当前只处理 `/doc/.../c/` 404 | 没有剥离 `/9.3.0-.../` 前缀 |

这意味着当前 mock 注入链路可能并没有稳定进入目标 editor HTML。若 URL 带版本前缀，`onlyofficeDesktopMock()` 拼出的文件路径会指向不存在的位置，注入会失败或落到其他 fallback。

### 结论 4：iframe 选择器 bug 是真实问题，但不是根因

当前 `src/lib/onlyoffice-editor.ts` 的 9.3 分支使用：

```ts
const iframeEl = (document.getElementById('iframe') as HTMLElement | null)
  ?.querySelector('iframe') as HTMLIFrameElement | null;
```

调试文档记录 runtime 中：

```text
document.getElementById('iframe') 返回 null
document.querySelector('iframe') 返回非 null
```

所以应该改成直接查找页面上的 iframe：

```ts
const iframeEl = document.querySelector('iframe') as HTMLIFrameElement | null;
```

但这个修复只会解决“拿不到 `__desktopApi`”的问题。拿到 API 后，如果仍传旧 `DOCY;v5` string 或解码后的旧 bytes，仍可能继续触发 `ConvertationOpenFormat`。

### 结论 5：旧文档里“Desktop Mock 已放弃”的判断需要被后续结论修正

旧的 `2026-05-31-onlyoffice-desktop-mode-mock.md` 曾写到 mock 方案因为 `Ec.Ms` 初始化不可达而放弃。

后续 `2026-06-14-onlyoffice-9.3.0-source-investigation.md` 修正了这一点：

- `Ec.Ms` 更像是错误的观察对象；
- Word 正确路径应关注 `nativeOpenFile()`、`jre()`、`ta.Ga`、`AscCommonWord.e2`；
- `code.js` 存在不等于一定需要服务器，socket.io 逻辑主要在 `sdk-all-min.js`；
- Desktop tarball + 正确 `AscDesktopEditor` mock 仍有理论可行性。

因此，文档体系里现在同时存在“放弃”和“重新评估”的结论。后续应以 2026-06-14 的 source investigation 和本复核记录为准。

## 四、当前最可能的失败链路

从现有证据看，当前失败不应被归为单点问题，而是至少有三层风险：

```text
页面创建 DocEditor
  -> api.js 生成带版本前缀的 editor URL
  -> Vite 当前未剥离版本前缀
  -> desktop mock 注入可能失败或不稳定
  -> 即使注入成功，onlyoffice-editor.ts 仍可能找不到 iframe
  -> 即使找到 iframe 并拿到 __desktopApi，传入数据仍是旧 DOCY string 或旧 decoded bytes
  -> 9.3 sdkjs 的 a9c / BRj 路径无法识别
  -> ConvertationOpenFormat / errorInconsistentExt
```

所以，下一步不能直接改 `empty_bin.ts`，也不能只改 iframe 选择器后宣称解决。必须逐层验证。

---

## 五、建议的验证顺序

### Step 1：先修正资源路径和 mock 注入链路

目标：确认 9.3 editor HTML 确实被 Vite 插件读取，并注入 `AscDesktopEditor` mock。

建议：

1. 恢复版本前缀剥离逻辑：

```ts
if (/^\/\d+\.\d+\.\d+-[a-f0-9]+/i.test(req.url)) {
  req.url = req.url.replace(/^\/\d+\.\d+\.\d+-[a-f0-9]+/i, '');
}
```

2. 或直接 patch `api.js`：

```js
const ver = '';
```

3. 添加明确日志：

```js
console.log('[DE] mock injected');
console.log('[DE] CreateEditorApi');
console.log('[DE] MOa patched -> BRj path active');
```

验证标准：

```text
[DE] mock injected
[DE] CreateEditorApi
[DE] MOa patched -> BRj path active
```

三条日志必须在同一次页面加载中出现。

### Step 2：修正 iframe 获取方式

目标：确保父页面能访问同源 iframe 的 `__desktopApi`。

建议修改：

```ts
const iframeEl = document.querySelector('iframe') as HTMLIFrameElement | null;
const api = (iframeEl?.contentWindow as any)?.__desktopApi;
```

验证标准：

```text
iframe exists: true
has __desktopApi: true
has asc_openDocumentFromBytes: true
```

### Step 3：记录每次打开文档的输入格式

目标：不要再猜 `asc_openDocumentFromBytes` 接收了什么。

建议在调用前记录：

```ts
console.log('[OO input]', {
  binDataType: typeof binData,
  binDataLength: typeof binData === 'string' ? binData.length : undefined,
  pendingCopyLength: pendingCopy.byteLength,
  pendingCopyHead: Array.from(pendingCopy.slice(0, 8)),
});
```

验证标准：

| 场景 | 期望看到 |
|------|----------|
| New Word 当前旧模板 | `binDataType: "string"`，`binDataLength` 约 9845，`pendingCopyHead` 非 `DOCY` |
| 已有 docx 文件 | 记录原始文件是否是 ZIP `PK`，以及是否经过 x2t 转换 |

### Step 4：用最小输入实验验证 `a9c` 假设

不要一次性改完整打开流程。先构造最小探针：

| 实验 | 输入 | 预期 |
|------|------|------|
| A | `binData` 原始 string | `a9c=false`，大概率失败 |
| B | decoded base64 bytes | `a9c=false`，大概率失败 |
| C | `TextEncoder().encode(binData)` | `a9c=true`，但可能因内部格式旧而后续失败 |
| D | 9.3 同代 empty template | 如果存在，才是有效候选 |

只有 C/D 的结果能判断“签名检查之后是否还存在版本兼容问题”。

### Step 5：再决定是走 Desktop Mock 还是最小 socket.io 协议

如果 Step 1-4 后仍卡在输入格式，则有两条路线：

| 路线 | 条件 | 成本 | 判断 |
|------|------|------|------|
| Desktop Mock | 能稳定触发 `CreateEditorApi` / `MOa patched`，且能找到 9.3 同代输入格式 | 中高 | 值得继续 |
| 最小 socket.io 协议 | Desktop Mock 仍绕不过文档加载协议或 native 数据路径 | 高 | 技术攻坚路线 |

## 六、当前不建议做的事

1. **不建议只改 iframe 选择器后直接判断成功/失败。**
   它只是修掉一个阻塞点，后面还有输入格式问题。

2. **不建议继续把旧 `DOCY;v5` base64 解码成 bytes 传给 9.3。**
   当前证据显示它会丢掉 envelope，且前 4 bytes 不满足 DOCY signature。

3. **不建议把 `TextEncoder().encode(DOCY string)` 当成最终修复。**
   它最多能通过 `a9c` 签名检查，不能证明旧内部格式能被 9.3 解析。

4. **不建议把问题归因成“缺少 wasm”。**
   wasm 是转换层，当前失败同时涉及 editor 初始化、mock 注入和内部文档格式。

5. **不建议直接切回 7.x 后删除 9.3 探索记录。**
   这些记录已经把 9.x 架构、Desktop 分支、输入格式风险拆清楚了，后续仍有复用价值。

## 七、建议沉淀到后续文档的统一表述

建议以后统一这样描述当前 9.3 升级状态：

> 9.3 升级失败不是单一资源缺失，而是 7.x standalone 架构到 9.x server/desktop 架构迁移导致的多层兼容问题。当前已确认 `sdkjs/web-apps` 可以升级到 9.3，但默认打开路径、Desktop mock 注入链路、`empty_bin.ts` 内部模板版本、`asc_openDocumentFromBytes` 输入格式必须同代匹配。旧 `DOCY;v5` 模板和 9.3 的文档打开路径不能直接混用。

## 八、短期行动清单

按优先级：

1. 修正 `onlyofficeVersionRewrite()` 或 `api.js`，确保 9.3 editor URL 能命中真实 HTML。
2. 在 `onlyofficeDesktopMock()` 注入脚本开头增加 `[DE] mock injected`。
3. 修正 `onlyoffice-editor.ts` iframe 查询方式。

5. 只跑 `New Word` 单一场景，收集 console 和 network 证据。
6. 根据结果决定是否提取 9.3 empty template 或转向 socket.io mock。
